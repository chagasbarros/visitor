Padrão de projeto

Padrão Visitor

Entendendo o padrão comportamental com a analogia do Auditor Fiscal — passo a passo, do zero.

1 Antes do código: a analogia

Sempre entenda o problema antes de programar.

🏢 Imagine o seguinte cenário

Uma cidade tem três tipos de empresa: Restaurante, Loja e Fábrica. A Receita Federal precisa enviar um Auditor Fiscal para cada uma delas para verificar impostos.


Cada empresa sabe receber o auditor (tem uma porta de entrada). Mas é o auditor quem decide o que fazer dentro de cada tipo de empresa — calcular o ISS em restaurantes, o ICMS em lojas, o IPI em fábricas.


Amanhã, a Receita pode criar um novo auditor (p. ex., de Vigilância Sanitária) sem precisar mexer em nenhuma empresa. Esse é o padrão Visitor.

🏢

Elemento

interface Empresa

Objeto que será visitado. Tem um método receberAuditor().

🍕

Elemento Concreto

class Restaurante

Implementa receberAuditor() chamando auditor.visitar(this).

🕵️

Visitor

interface Auditor

Declara um método visitar() para cada tipo de empresa.

📋

Visitor Concreto

class AuditorFiscal

Implementa a lógica real de fiscalização para cada empresa.

2 Criando as interfaces

Primeiro definimos os "contratos" — o que cada parte deve ser capaz de fazer.

Empresa.java & Auditor.java 
// ─────────────────────────────────────────────────
// PASSO 1: Interface do ELEMENTO (as empresas)
// Toda empresa deve ter uma "porta de entrada" 
// para receber qualquer auditor.
// ─────────────────────────────────────────────────

public interface Empresa {
    // Cada empresa deve saber "receber" um auditor
    void receberAuditor(Auditor auditor);
}


// ─────────────────────────────────────────────────
// PASSO 2: Interface do VISITOR (o auditor)
// O auditor declara que sabe visitar cada tipo 
// de empresa que existe.
// ─────────────────────────────────────────────────

public interface Auditor {
    void visitar(Restaurante restaurante); // sabe visitar restaurante
    void visitar(Loja loja);                 // sabe visitar loja
    void visitar(Fabrica fabrica);           // sabe visitar fábrica
}
💬 Por que duas interfaces? A Empresa representa a estrutura estável (as empresas não mudam). O Auditor representa as operações que podem ser criadas e multiplicadas no futuro, sem tocar nas empresas.

3 Implementando as empresas

Cada empresa implementa o método receberAuditor(). Note que todas fazem a mesma coisa: chamam auditor.visitar(this).

Restaurante.java · Loja.java · Fabrica.java 
// ──────────────────────────────────────────
public class Restaurante implements Empresa {

    private String nome;
    private double faturamentoMensal;

    public Restaurante(String nome, double faturamentoMensal) {
        this.nome = nome;
        this.faturamentoMensal = faturamentoMensal;
    }

    public String  getNome()              { return nome; }
    public double getFaturamentoMensal() { return faturamentoMensal; }

    @Override
    public void receberAuditor(Auditor auditor) {
        // "Olá auditor, eu sou um Restaurante, pode entrar!"
        // O Java escolhe automaticamente o método visitar(Restaurante)
        auditor.visitar(this);
    }
}

// ──────────────────────────────────────────
public class Loja implements Empresa {

    private String nome;
    private double valorEstoque;

    public Loja(String nome, double valorEstoque) {
        this.nome = nome;
        this.valorEstoque = valorEstoque;
    }

    public String  getNome()         { return nome; }
    public double getValorEstoque() { return valorEstoque; }

    @Override
    public void receberAuditor(Auditor auditor) {
        auditor.visitar(this); // chama visitar(Loja)
    }
}

// ──────────────────────────────────────────
public class Fabrica implements Empresa {

    private String nome;
    private int    qtdFuncionarios;
    private double valorProducaoMensal;

    public Fabrica(String nome, int qtdFuncionarios, double valorProducaoMensal) {
        this.nome = nome;
        this.qtdFuncionarios = qtdFuncionarios;
        this.valorProducaoMensal = valorProducaoMensal;
    }

    public String  getNome()               { return nome; }
    public int     getQtdFuncionarios()    { return qtdFuncionarios; }
    public double  getValorProducaoMensal(){ return valorProducaoMensal; }

    @Override
    public void receberAuditor(Auditor auditor) {
        auditor.visitar(this); // chama visitar(Fabrica)
    }
}

4 Criando o Auditor Fiscal

Aqui fica toda a lógica de negócio. O auditor sabe exatamente o que fazer em cada tipo de empresa.

AuditorFiscal.java 
public class AuditorFiscal implements Auditor {

    // ── Restaurantes pagam ISS: 5% sobre faturamento ──
    @Override
    public void visitar(Restaurante r) {
        double iss = r.getFaturamentoMensal() * 0.05;
        System.out.println(
            "🍕 Restaurante: " + r.getNome() +
            "\n   ISS devido (5%): R$ " + iss
        );
    }

    // ── Lojas pagam ICMS: 12% sobre o valor do estoque ──
    @Override
    public void visitar(Loja l) {
        double icms = l.getValorEstoque() * 0.12;
        System.out.println(
            "🛒 Loja: " + l.getNome() +
            "\n   ICMS devido (12%): R$ " + icms
        );
    }

    // ── Fábricas pagam IPI: 10% sobre produção + R$200/funcionário ──
    @Override
    public void visitar(Fabrica f) {
        double ipi  = f.getValorProducaoMensal() * 0.10;
        double inss = f.getQtdFuncionarios() * 200.0;
        System.out.println(
            "🏭 Fábrica: " + f.getNome() +
            "\n   IPI  devido (10%): R$ " + ipi +
            "\n   INSS devido:        R$ " + inss
        );
    }
}
🔑 Ponto-chave: toda a lógica de impostos fica aqui, e não espalhada dentro de cada empresa. As classes Restaurante, Loja e Fabrica não sabem nada sobre cálculo de impostos — só sabem abrir a porta.

5 Colocando tudo para rodar

O método main é onde criamos as empresas, o auditor, e mandamos ele fazer a ronda.

Main.java 
import java.util.List;

public class Main {
    public static void main(String[] args) {

        // 1. Criamos as empresas da cidade
        List<Empresa> empresas = List.of(
            new Restaurante("Sabor da Terra",  50_000.0),
            new Loja       ("Moda & Estilo",   80_000.0),
            new Fabrica    ("AçoForte Ltda", 30, 200_000.0)
        );

        // 2. Criamos o auditor fiscal
        Auditor auditorFiscal = new AuditorFiscal();

        // 3. Mandamos o auditor visitar cada empresa
        System.out.println("=== FISCALIZAÇÃO DA RECEITA FEDERAL ===");
        for (Empresa e : empresas) {
            e.receberAuditor(auditorFiscal); // cada empresa "abre a porta"
            System.out.println("───────────────────────────────────────");
        }
    }
}

6 O que acontece por baixo dos panos?

Esse "caminho secreto" que a chamada percorre tem nome: Double Dispatch.

1

O loop chama e.receberAuditor(auditorFiscal)

Neste ponto, e pode ser qualquer tipo de empresa. O Java sabe em tempo de execução qual classe concreta está guardada ali.

2

A empresa chama auditor.visitar(this)

O this aqui é o tipo concreto, por exemplo Restaurante. O Java agora sabe o tipo exato do objeto.

3

O Java escolhe o método visitar(Restaurante)

Como o tipo é conhecido, o método certo do AuditorFiscal é chamado automaticamente. Isso é o double dispatch: dois despachos dinâmicos em cadeia.

4

A lógica do auditor é executada para aquele tipo

O cálculo de ISS roda para restaurante, ICMS para loja, IPI/INSS para fábrica — sem nenhum if/else ou instanceof!

7 Adicionando um novo auditor sem mexer em nada

Essa é a grande vantagem. Quer um Auditor de Vigilância Sanitária? Crie só uma nova classe.

AuditorSanitario.java — NOVO, sem alterar nada 
public class AuditorSanitario implements Auditor {

    @Override
    public void visitar(Restaurante r) {
        System.out.println("🔬 Verificando cozinha de: " + r.getNome());
        System.out.println("   → Inspeção de alimentos obrigatória!");
    }

    @Override
    public void visitar(Loja l) {
        System.out.println("🔬 Verificando produtos de: " + l.getNome());
        System.out.println("   → Checar prazo de validade.");
    }

    @Override
    public void visitar(Fabrica f) {
        System.out.println("🔬 Vistoria de segurança em: " + f.getNome());
        System.out.println("   → EPI e saídas de emergência.");
    }
}

// No main, basta trocar o auditor:
Auditor sanitario = new AuditorSanitario();
empresas.forEach(e -> e.receberAuditor(sanitario));
// As empresas não foram modificadas em nenhum momento!

8 Resumo: quando usar (e quando não usar)

Padrões de projeto existem para resolver problemas. Conheça o contexto certo de aplicação.

Situação Visitor? Motivo
Precisa de várias operações sobre uma hierarquia estável ✅ Sim É exatamente o ponto forte do padrão
Quer adicionar operações sem modificar as classes existentes ✅ Sim Segue o princípio Open/Closed
A hierarquia de tipos muda com frequência ❌ Não Cada novo tipo exige atualizar todos os visitors
Você só tem uma operação simples ❌ Não Complexidade desnecessária; use polimorfismo simples
